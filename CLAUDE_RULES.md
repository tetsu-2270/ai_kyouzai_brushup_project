# Claude Code 開発ルール（プロジェクト固有・Claude Code向け手順）

このプロジェクトをClaude Codeで実装する際の、Claude Code固有の実装手順・完了報告方法・権限モード注記をまとめたファイルです。

全開発共通ルール（役割分担・確認方針・即席シェルコマンド・正式な検証入口の一般原則・完了レポート[主確認手段]・エビデンス[補助確認手段]・ローカル開発の合否基準・Git安全ルール）は`~/ai-development-rules/DEVELOPMENT_RULES.md`を正とします。このプロジェクト固有の仕様（目的・output構成・OCR前提・Git管理方針・必読設計書・検証入口の具体値・禁止事項）は[`PROJECT_RULES.md`](PROJECT_RULES.md)を正とします。**このファイルには、それらへ書くべき内容を重複記載しません。**

`~/ai-development-rules/DEVELOPMENT_RULES.md`が存在しない環境でも、このファイルと`PROJECT_RULES.md`だけで安全に作業できることを前提にしています。

## 基本方針

- 実装前に既存ファイル・既存の設計書・README・タスク一覧を確認し、矛盾しないように実装する。
- 1回の作業範囲を小さく区切る。
- 仕様変更が必要な場合は、理由と影響範囲を説明して承認を得る（詳細は`PROJECT_RULES.md`「8. 既存仕様を壊さないこと」参照）。

## 進め方

- `docs/05_implementation_tasks.md` の順番に進める。
- 1タスクごとに完了報告する。
- 変更ファイル、実施内容、テスト結果を必ず報告する。
- 大きな変更を一度に行わない。破壊的変更を行う前に確認する。

## 実行確認の運用ルール

確認を減らす一般方針・事前確認が必要な操作・即席シェルコマンドの書き方は`~/ai-development-rules/DEVELOPMENT_RULES.md`「2. 確認を減らす」「3. 即席シェルコマンド」を正とします。

### Claude Codeの権限モードに関する注記

- Claude Codeの通常モードでは、ツール実行時（特にBashコマンド）に確認ダイアログが出る場合がある。これはClaude Code本体の権限モード・安全機構によるものであり、上記「実行確認について」のルールとは別物である。
- 確認を減らしたい場合は、Claude Code側で`acceptEdits`モードを使う。
- `acceptEdits`でも、すべてのBashコマンド確認が消えるとは限らない。
- 危険な権限バイパスモード（`--dangerously-skip-permissions`等）は通常運用では推奨しない。
- Claude Code側の権限確認ダイアログが出た場合は、ユーザーが承認する必要がある。
- ただし、Claude Code自身は「実行してよいですか」といった作業ごとの確認質問をしない。権限確認が出そうな作業は、開始前にまとめて宣言し（例:「この作業ではpytest実行・run_sample.sh実行を行います。Bash実行時に確認ダイアログが出る可能性があります」）、以降は個別に確認せず進める。

## Claude Code運用ルール（指示文・報告・確認コマンドの分離。重要）

今後の開発作業で、指示・報告・確認・Git操作が混ざって手間が増えないようにするための運用ルールです。**機能追加ではなく作業の進め方そのものに関するルール**であり、以降のすべての作業に適用します。

### 1. 作業は一つずつ進める

- まず現状が本当に動くかを確認する。
- 動いていない箇所を最小単位で直す。
- その修正が完了したかを確認する。
- その後で初めて、構築自動化・AWS・Dockerなど新しい設計拡張に広げる。
- 現在の検証不足がある場合は、新しい設計拡張よりも検証を優先する。
- **仕様を不用意に広げない。** 依頼されていない機能・抽象化・将来のための拡張を先回りして追加しない。

### 2. Claude Code向け指示文とユーザー側コマンドを混ぜない

- Claude Code向け指示文には、実装内容・テスト・docs更新・完了報告形式を書く。
- ユーザー側の一発確認コマンドは、原則としてClaude Code向け指示文に埋め込まない。
- ユーザー側確認コマンドが必要な場合は、Claude Code向け指示文とは別に提示する。
- Git commit / tag / push 用コマンドも、実装指示文とは別に提示する。
- 実インストール（Homebrew/Tesseract等）やPATH反映など、ユーザーの端末で手動実行が必要なものも別コマンドとして扱う。

### 3. Claude Code向け指示文には実行方針を必ず入れる

Claude Code向けの実装指示文には、毎回以下の趣旨を含める（`CLAUDE_RULES.md参照`とだけ書いて済ませず、指示文自体に短い実行方針ブロックを含める）。

```text
- できるだけ確認せずに進める。
- 通常の調査・編集・テスト実行は自律的に進める。
- ただし、Git操作・認証・有料サービス・破壊的変更・秘密情報操作・実インストール・
  ユーザーにしかできない作業は事前確認する。
```

### 4. 完了報告は一発コピペできる形式にする（Claude Code完了レポート＝主確認手段。HTML Artifactで出力）

Claude Codeの完了報告は、Codexが追加でファイルを読まなくても判断できる**自己完結した完了レポート**とする（`~/ai-development-rules/DEVELOPMENT_RULES.md`「6. Claude Code完了レポート（主確認手段）」参照。`logs/evidence/`等のエビデンスは補助確認手段）。以下のMarkdownテンプレートを既定とし、**`python3 -m src.completion_report`でコピー用ボタン付きの自己完結型HTML Artifactへ変換して出力する**（`PROJECT_RULES.md`「9.1 完了レポートのHTML Artifact化と保存先」参照。新規のArtifact生成スクリプトを都度手作業で組み立てない）。チャット本文では、判定・Artifact保存先（`output/reports/`のタイムスタンプ付き・latest両方）・エビデンス保存先だけを短く報告し、レポート全文を重複して貼らない。

```text
# Claude Code実装完了レポート

## 1. 判定
- 完了 / 条件付き完了 / 未完了
- 判定理由:

## 2. 対応概要
<何を実装・修正したか>

## 3. 変更ファイル
- <ファイル>: <変更内容>

## 4. 実装内容
<主要な実装内容>

## 5. 設計上の判断
- <判断内容と理由>

## 6. テスト
- 正式な検証入口:
- 総合結果:
- 終了コード:
- 総テスト数:
- 成功:
- 失敗:
- スキップ:
- サンプル・生成物確認:

## 7. 実データ確認
- 対象:
- 実行方法:
- 確認結果:

## 8. 実装前後比較
| 指標 | 実装前 | 実装後 |
|---|---:|---:|
| <指標> | <値> | <値> |

## 9. 完了条件との対応
- [x] <達成した条件>
- [ ] <未達の条件と理由>

## 10. 残課題・制限事項
- <残課題>
- なしの場合は「なし」

## 11. 既知の不具合・懸念
- <不具合または懸念>
- なしの場合は「なし」

## 12. エビデンス
- 最新索引:
- run_id:
- summary:
- manifest:

## 13. ドキュメント
- docs/05_implementation_tasks.md:
- docs/99_*:
- その他:

## 14. Git操作

- 無い場合は「なし」
```

品質要件（成功だけでなく残課題も書く・未完了は未完了と書く・実データの不一致を省略しない・完了条件ごとに達成/未達を対応付ける・「問題なし」には根拠を示す・推測と事実を区別する・長い生ログを貼らない・Codexが追加調査すべき箇所を明示する）は`~/ai-development-rules/DEVELOPMENT_RULES.md`「6.3 品質要件」を正とする。「6. テスト」「12. エビデンス」には、正式な検証入口（`PROJECT_RULES.md`「9. このプロジェクトの正式な検証入口とエビデンス保存先」参照）を実行した結果とエビデンス保存先（`logs/evidence/<run_id>/`）を記載し、長い実行ログそのものは貼らない。

### 5. ユーザー側確認スクリプトを出す場合の形式

ユーザー側確認スクリプト（一発コピペ用の確認コマンド）が必要な場合は、以下の形式にする。

- Claude Code向け指示文とは分けて提示する。
- 「これはユーザー側コンソール用」と明記する。
- 一発コピペできる形にする。
- 判定OKの理由を出す。
- 判定NGの理由を出す。
- 最終判定として「判定: OK」または「判定: NG」を出す。
- NGの場合は非ゼロ終了する（呼び出し元がスクリプトの成否を機械的に判定できるようにする）。

### 6. Git操作コマンドの扱い

- Git commit / tag / push はClaude Codeに実行させない。
- Git保存が必要な場合は、ユーザー側手動コマンドとして別途提示する。
- Git保存前には `git status` と `git diff --stat` を確認する（内容はClaude Codeが事前に提示してよい）。
- Git管理対象外にすべきもの・commit前の確認事項は`~/ai-development-rules/DEVELOPMENT_RULES.md`「9. Git・安全ルール」を参照。このプロジェクトでの追加対象（`logs/evidence/`等）は`PROJECT_RULES.md`「6. input・output・logs・生成物のGit管理方針」を参照。

## 実装品質

- Pythonは型ヒントを付ける。
- 関数は責務を小さくする。
- 入力データのバリデーションを行う。
- 例外時に原因が分かるエラーメッセージを出す。
- ファイル入出力はUTF-8を前提にする。
- 日本語テキストを扱うため、文字化けに注意する。

## テスト

- 主要処理にはテストを追加する。
- 既存テストがある場合は壊さない。
- 実装後に最低限以下を確認する。
  - サンプルJSONが読み込める
  - ブラッシュアップMarkdownが出力できる
  - Canva向け設計Markdownが出力できる
  - テストが通る
- 正式な検証入口・完了レポート・エビデンスの扱いは`PROJECT_RULES.md`「9. このプロジェクトの正式な検証入口とエビデンス保存先」・`~/ai-development-rules/DEVELOPMENT_RULES.md`「5. 正式な検証入口」「6. Claude Code完了レポート（主確認手段）」「7. エビデンス（補助確認手段）」「8. ローカル開発の合否基準」を参照。
