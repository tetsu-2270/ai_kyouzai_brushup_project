# Claude Code 開発ルール

このプロジェクトをClaude Codeで実装する際の共通ルールです。

## 基本方針

- 勝手に仕様を変更しない。
- 仕様変更が必要な場合は、理由と影響範囲を説明して承認を得る。
- 実装前に既存ファイルを確認する。
- 既存の設計書、README、タスク一覧と矛盾しないように実装する。
- 1回の作業範囲を小さく区切る。

## 進め方

- `docs/05_implementation_tasks.md` の順番に進める。
- 1タスクごとに完了報告する。
- 変更ファイル、実施内容、テスト結果を必ず報告する。
- 大きな変更を一度に行わない。
- 破壊的変更を行う前に確認する。

## 実行確認の運用ルール（重要）

このプロジェクトに限らず、今後すべての作業に適用する運用ルールです。ユーザーは、通常の作業ごとに毎回確認されることを望んでいません。ただし、Claude Code本体の権限モードや安全機構により、ツール実行時に確認ダイアログが表示される場合があります（詳細は後述「Claude Codeの権限モードに関する注記」）。この運用ルールは、Claude Code自身が作業ごとに確認質問を作らないようにするためのものであり、Claude Code本体が出す権限確認ダイアログそのものを無くすものではありません。

### 1. 確認を求めない範囲

以下は、ユーザーに都度確認せず進める。

- ファイル編集
- ドキュメント更新
- テスト追加
- `pytest`実行
- `scripts/run_sample.sh`実行
- ローカルのサンプル生成
- `output/`配下への生成
- `archive/`や`docs/`配下の整理
- 指示範囲内の通常作業

不要な確認の例:
- このファイルを修正してよいですか？
- pytestを実行してよいですか？
- run_sample.shを実行してよいですか？
- ドキュメントを更新してよいですか？
- 次に進めてよいですか？

### 2. 権限確認が出そうな操作は事前にまとめて宣言する

Claude Code本体の権限確認が出る可能性がある場合（例: Bashでのローカルコマンド実行を伴う作業）は、作業開始前に何を行うかをまとめて宣言する。

例: 「この作業では、ファイル編集、pytest実行、run_sample.sh実行、サンプル画像生成のためのローカルコマンド実行を行います。Claude Codeの権限モードによっては、Bash実行時に確認ダイアログが出る可能性があります。確認が出た場合は承認してください。」

この宣言をしたら、個別の作業ごとに「実行してよいですか？」とは聞かず、そのまま進める。

### 3. 事前確認が必要な操作

以下に該当する場合だけ、作業開始前に明示する。

- パスワード入力が必要
- ログインが必要
- 認証コード入力が必要
- APIキーやトークン入力が必要
- 有料サービスの利用・課金が発生する可能性がある
- 外部サービスへの送信が発生する
- GitHub pushなど、外部環境へ反映する操作がある
- ローカルアプリの起動や手動操作が必要
- ユーザーのファイル選択やアップロードが必要
- ユーザー本人にしかできない承認操作が必要
- Claude Codeの権限確認が必ず出る可能性が高いBashコマンド

### ユーザー作業が必要な場合の扱い

- **事前に済ませられる場合**: 作業開始前に必要作業を明示する。
  - 例: 「この作業ではGoogle認証が必要です。先にログインを完了してください。」
  - 例: 「この作業ではGitHubのPATが必要です。環境変数に設定してください。」
  - 例: 「この作業では入力元画像を`input/source/`に置いてください。」
- **途中で必ずユーザー作業が入る場合**: 作業開始前に、途中でユーザー作業が入ることを明示する。
  - 例: 「途中でmacOSの権限確認ダイアログが出る可能性があります。」
  - 例: 「PDF変換時に追加ツールのインストール確認が出る可能性があります。」
  - 例: 「外部API利用時に認証コード入力が必要になる可能性があります。」

### 原則

- 通常のファイル編集、テスト実行、サンプル実行、ドキュメント更新は確認なしで進める。
- Claude Code本体の権限確認が出そうな作業は、開始前にまとめて宣言し、以降は個別に確認しない。
- ユーザー本人の判断・認証・課金・外部反映が必要な場合だけ事前に明示する。
- 作業途中で止まる可能性があるものは、開始前にその可能性を説明する。
- 作業完了後に、実施内容・結果・テスト結果・残課題を報告する。

### Claude Codeの権限モードに関する注記

- Claude Codeの通常モードでは、ツール実行時に確認が出る場合がある。
- 確認を減らしたい場合は、Claude Code側で`acceptEdits`モードを使う。
- `acceptEdits`でも、すべてのBashコマンド確認が消えるとは限らない。
- 危険な権限バイパスモードは通常運用では推奨しない。
- Claude Code側で確認が出た場合は、ユーザーが承認する必要がある。
- ただし、Claude Code自身は作業ごとの確認質問をしない（本ルールが適用される）。

## Git運用

- 可能な範囲で、1タスク完了ごとにコミットする。
- コミットメッセージは日本語で簡潔に書く。
- 例：`教材ページのデータモデルを追加`
- 失敗した変更を残さない。
- 不要な一時ファイル、キャッシュ、ログはコミットしない。

## 実装品質

- Pythonは型ヒントを付ける。
- 関数は責務を小さくする。
- 入力データのバリデーションを行う。
- 例外時に原因が分かるエラーメッセージを出す。
- ファイル入出力はUTF-8を前提にする。
- 日本語テキストを扱うため、文字化けに注意する。

## テスト

- 主要処理にはテストを追加する。
- 既存テストがある場合は壊さない。
- 実装後に最低限以下を確認する。
  - サンプルJSONが読み込める
  - ブラッシュアップMarkdownが出力できる
  - Canva向け設計Markdownが出力できる
  - テストが通る

## ドキュメント

- 実装内容が変わったらREADME.mdを更新する。
- 入力形式が変わったら `docs/03_data_format.md` を更新する。
- 出力形式が変わったら `docs/04_output_spec.md` を更新する。
- タスク完了状況が変わったら `docs/05_implementation_tasks.md` を更新する。

### ドキュメント整合性の運用ルール（重要）

- コード・CLI・出力仕様・テストに変更を加えた場合は、必ず `docs/05_implementation_tasks.md` のPhase/Task進捗を確認し、必要に応じて更新する。
- 仕様変更や新機能追加を行った場合は、README.mdおよび `docs/01_requirements.md`〜`docs/04_output_spec.md` の該当箇所に矛盾がないか確認する。
- `docs/99_*` は時点スナップショット文書として扱い、既存ファイルを上書きしない。
- 新しいレビュー結果や実装状態スナップショットを残す場合は、日付またはバージョン付きの新規ファイル（例: `docs/99_review_2026-08.md`）として追加する。
- `docs/99_*` の内容は作成時点の情報であり、現行仕様の正はREADME.md・`docs/01_requirements.md`〜`docs/04_output_spec.md`・実コード・pytest結果とする。
- 作業完了報告には、`docs/05_implementation_tasks.md` を更新したか・更新不要と判断したか、および`docs/99_*` を新規追加したか・追加不要と判断したかを必ず含める。

## 禁止事項

- ユーザー承認なしに技術スタックを大きく変更しない。
- 不要に複雑な構成にしない。
- 外部API前提にしない。
- 有料サービス前提にしない。
- 元教材の意図を無視して文章を大きく改変しない。
- Canvaで再現できないレイアウトを前提にしない。

## プロジェクト設計ルール（output構成・editable中間ファイル・source情報の扱い）

Phase 9.2までに確定した、このプロジェクトの共通設計ルールです。**今後のPhaseで変更する場合は、「基本方針」のとおり理由と影響範囲を説明して承認を得てから変更してください。** 詳細な仕様は[`docs/04_output_spec.md`](docs/04_output_spec.md)を正とし、ここではプロジェクト標準として要点のみ固定化します。

### 1. output構成

`build-all`が生成する`output/`配下は、以下の5ディレクトリに役割を分ける。

```text
output/
  editable/
    lesson_pages.json      # 正式な編集対象（再生成時にユーザーが編集するのはここのみ）
  rendered/
    page_NNN.png           # 正式な完成画像
  canva/
    canva_design.md        # 正式なCanva指示書（オプション出力。主outputではない）
  exports/
    material.md / .docx / .pdf / .pptx   # 正式な完成output
  compat/
    lesson_pages.json / canva_design.md / brushup.md / brushup.docx / brushup.pdf
                            # Phase 8以前の旧仕様との互換用output。新規利用では通常参照しない
```

各ディレクトリの役割:

- `editable/`: 再生成時にユーザーが編集する正式な中間output。
- `rendered/`: 配布・確認用の完成画像。
- `canva/`: 正式なCanva指示書。数ある完成output形式の一つであり、主outputではない。
- `exports/`: 正式な完成output（Markdown/DOCX/PDF/PPTX）。
- `compat/`: Phase 8以前の旧仕様との互換用output。`--no-compat-output`で生成自体を無効化できる。新規利用では通常参照しない。
- `output_dir`直下には、通常ユーザーが使う完成outputを置かない（`scenario/`・`review_report.md`・`imported_pages.json`・`assets/`は正式outputとの役割重複が無いため直下のまま）。

### 2. editable中間ファイルの扱い

`output/editable/lesson_pages.json`は、ユーザーが編集する正式な中間outputである。**完成画像・PDF・DOCX・PPTXを直接編集するのではなく**、以下の運用をプロジェクト標準とする。

```text
output/editable/lesson_pages.json を編集
↓
regenerate
↓
rendered/ や exports/ を再生成
```

### 3. source情報の扱い

- 元資料由来の画像・ページ画像・source assetsを落とさない。PDF・PPTX・画像などから取り込んだ元画像・ページ画像は、可能な限り`source_image`/`source_assets`（既存設計）として保持する。
- `source_page_no`は内部メタデータとして保持する。通常、購入者・受講者向けの完成output（`rendered/`・`exports/`・`canva/`・DOCX/PDF/動画シナリオ）には表示しない。制作者が確認したい場合は`review-report`コマンドを使う。

### 4. 今後のPhase指示文での参照方法

今後のPhase指示文では、このルールを毎回書き下す代わりに、以下のように短く参照してよい。

```text
CLAUDE_RULES.md および docs/04_output_spec.md に定義済みの
output構成・editable中間ファイル・source情報の扱いを維持してください。
今回のPhaseでは、それらの既存設計を変更しないでください。
```

## 重要な前提

このシステムの目的は、教材画像・教材ページの内容をもとに、以下を作ることです。

1. 教材本文のブラッシュアップ案
2. ページ構成の改善案
3. Canvaで再現しやすい画像・レイアウト設計書
4. 最終的に人間が確認・修正しやすいMarkdown出力

完全自動化よりも、まずは人間が確認しやすい中間成果物を安定して出すことを優先します。
