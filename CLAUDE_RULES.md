# Claude Code 開発ルール

このプロジェクトをClaude Codeで実装する際の共通ルールです。

## 基本方針

- 勝手に仕様を変更しない。
- 仕様変更が必要な場合は、理由と影響範囲を説明して承認を得る。
- 実装前に既存ファイルを確認する。
- 既存の設計書、README、タスク一覧と矛盾しないように実装する。
- 1回の作業範囲を小さく区切る。

## 進め方

- `docs/05_implementation_tasks.md` の順番に進める。
- 1タスクごとに完了報告する。
- 変更ファイル、実施内容、テスト結果を必ず報告する。
- 大きな変更を一度に行わない。
- 破壊的変更を行う前に確認する。

## 実行確認の運用ルール（重要）

このプロジェクトに限らず、今後すべての作業に適用する運用ルールです。ユーザーは、通常の作業ごとに毎回確認されることを望んでいません。ただし、Claude Code本体の権限モードや安全機構により、ツール実行時に確認ダイアログが表示される場合があります（詳細は後述「Claude Codeの権限モードに関する注記」）。この運用ルールは、Claude Code自身が作業ごとに確認質問を作らないようにするためのものであり、Claude Code本体が出す権限確認ダイアログそのものを無くすものではありません。

### 1. 確認を求めない範囲

以下は、ユーザーに都度確認せず進める。

- ファイル編集
- ドキュメント更新
- テスト追加
- `pytest`実行
- `scripts/run_sample.sh`実行
- ローカルのサンプル生成
- `output/`配下への生成
- `archive/`や`docs/`配下の整理
- 指示範囲内の通常作業

不要な確認の例:
- このファイルを修正してよいですか？
- pytestを実行してよいですか？
- run_sample.shを実行してよいですか？
- ドキュメントを更新してよいですか？
- 次に進めてよいですか？

### 2. 権限確認が出そうな操作は事前にまとめて宣言する

Claude Code本体の権限確認が出る可能性がある場合（例: Bashでのローカルコマンド実行を伴う作業）は、作業開始前に何を行うかをまとめて宣言する。

例: 「この作業では、ファイル編集、pytest実行、run_sample.sh実行、サンプル画像生成のためのローカルコマンド実行を行います。Claude Codeの権限モードによっては、Bash実行時に確認ダイアログが出る可能性があります。確認が出た場合は承認してください。」

この宣言をしたら、個別の作業ごとに「実行してよいですか？」とは聞かず、そのまま進める。

### 3. 事前確認が必要な操作

以下に該当する場合だけ、作業開始前に明示する。

- パスワード入力が必要
- ログインが必要
- 認証コード入力が必要
- APIキーやトークン入力が必要
- 有料サービスの利用・課金が発生する可能性がある
- 外部サービスへの送信が発生する
- GitHub pushなど、外部環境へ反映する操作がある
- ローカルアプリの起動や手動操作が必要
- ユーザーのファイル選択やアップロードが必要
- ユーザー本人にしかできない承認操作が必要
- Claude Codeの権限確認が必ず出る可能性が高いBashコマンド

### ユーザー作業が必要な場合の扱い

- **事前に済ませられる場合**: 作業開始前に必要作業を明示する。
  - 例: 「この作業ではGoogle認証が必要です。先にログインを完了してください。」
  - 例: 「この作業ではGitHubのPATが必要です。環境変数に設定してください。」
  - 例: 「この作業では入力元画像を`input/source/`に置いてください。」
- **途中で必ずユーザー作業が入る場合**: 作業開始前に、途中でユーザー作業が入ることを明示する。
  - 例: 「途中でmacOSの権限確認ダイアログが出る可能性があります。」
  - 例: 「PDF変換時に追加ツールのインストール確認が出る可能性があります。」
  - 例: 「外部API利用時に認証コード入力が必要になる可能性があります。」

### 原則

- 通常のファイル編集、テスト実行、サンプル実行、ドキュメント更新は確認なしで進める。
- Claude Code本体の権限確認が出そうな作業は、開始前にまとめて宣言し、以降は個別に確認しない。
- ユーザー本人の判断・認証・課金・外部反映が必要な場合だけ事前に明示する。
- 作業途中で止まる可能性があるものは、開始前にその可能性を説明する。
- 作業完了後に、実施内容・結果・テスト結果・残課題を報告する。

### Claude Codeの権限モードに関する注記

- Claude Codeの通常モードでは、ツール実行時に確認が出る場合がある。
- 確認を減らしたい場合は、Claude Code側で`acceptEdits`モードを使う。
- `acceptEdits`でも、すべてのBashコマンド確認が消えるとは限らない。
- 危険な権限バイパスモードは通常運用では推奨しない。
- Claude Code側で確認が出た場合は、ユーザーが承認する必要がある。
- ただし、Claude Code自身は作業ごとの確認質問をしない（本ルールが適用される）。

## Claude Code運用ルール（指示文・報告・確認コマンドの分離。重要）

今後の開発作業で、指示・報告・確認・Git操作が混ざって手間が増えないようにするための運用ルールです。**機能追加ではなく作業の進め方そのものに関するルール**であり、以降のすべての作業に適用します。

### 1. 作業は一つずつ進める

- まず現状が本当に動くかを確認する。
- 動いていない箇所を最小単位で直す。
- その修正が完了したかを確認する。
- その後で初めて、構築自動化・AWS・Dockerなど新しい設計拡張に広げる。
- 現在の検証不足がある場合は、新しい設計拡張よりも検証を優先する。
- **仕様を不用意に広げない。** 依頼されていない機能・抽象化・将来のための拡張を先回りして追加しない。

### 2. Claude Code向け指示文とユーザー側コマンドを混ぜない

- Claude Code向け指示文には、実装内容・テスト・docs更新・完了報告形式を書く。
- ユーザー側の一発確認コマンドは、原則としてClaude Code向け指示文に埋め込まない。
- ユーザー側確認コマンドが必要な場合は、Claude Code向け指示文とは別に提示する。
- Git commit / tag / push 用コマンドも、実装指示文とは別に提示する。
- 実インストール（Homebrew/Tesseract等）やPATH反映など、ユーザーの端末で手動実行が必要なものも別コマンドとして扱う。

### 3. Claude Code向け指示文には実行方針を必ず入れる

Claude Code向けの実装指示文には、毎回以下の趣旨を含める（`CLAUDE_RULES.md参照`とだけ書いて済ませず、指示文自体に短い実行方針ブロックを含める）。

```text
- できるだけ確認せずに進める。
- 通常の調査・編集・テスト実行は自律的に進める。
- ただし、Git操作・認証・有料サービス・破壊的変更・秘密情報操作・実インストール・
  ユーザーにしかできない作業は事前確認する。
```

### 4. 完了報告は一発コピペできる形式にする

完了時の報告内容は、曖昧な番号付きリストではなく、ユーザーがそのまま貼り付けられる完成報告形式にする。以下のテンプレートを既定とする。

```text
# 実装完了報告：<Phase名または修正名>

## 1. 対応概要
<何を直したか>

## 2. 原因
<原因がある場合は記載。不明な場合は不明と明記>

## 3. 変更ファイル
- <file>: <変更内容>
- <file>: <変更内容>

## 4. 実装内容
<具体的な修正内容>

## 5. 成功判定・失敗判定の変更
<正常終了/異常終了の条件をどう変えたか。変更が無ければその旨を明記>

## 6. 実行したコマンド
​```bash
<実行したコマンド>
​```

## 7. テスト結果
​```text
<pytest結果、run_sample結果など>
​```

## 8. 実データ確認結果
​```text
<input/source など実データで確認した結果。対象外ならその旨を明記>
​```

## 9. ログ・出力例
​```text
<重要なログ、生成ファイル数、出力例。該当なければ「該当なし」>
​```

## 10. 未対応・制限事項
- <あれば記載>
- なし

## 11. Git操作
Git commit / tag / push は未実施です。
```

### 5. ユーザー側確認スクリプトを出す場合の形式

ユーザー側確認スクリプト（一発コピペ用の確認コマンド）が必要な場合は、以下の形式にする。

- Claude Code向け指示文とは分けて提示する。
- 「これはユーザー側コンソール用」と明記する。
- 一発コピペできる形にする。
- 判定OKの理由を出す。
- 判定NGの理由を出す。
- 最終判定として「判定: OK」または「判定: NG」を出す。
- NGの場合は非ゼロ終了する（呼び出し元がスクリプトの成否を機械的に判定できるようにする）。

### 6. Git操作コマンドの扱い

- Git commit / tag / push はClaude Codeに実行させない。
- Git保存が必要な場合は、ユーザー側手動コマンドとして別途提示する。
- Git保存前には `git status` と `git diff --stat` を確認する（内容はClaude Codeが事前に提示してよい）。
- `input/`・`output/`・`logs/*.log`・認証情報・秘密情報を誤ってGit追加しない。

## Git運用

- 可能な範囲で、1タスク完了ごとにコミットする。
- コミットメッセージは日本語で簡潔に書く。
- 例：`教材ページのデータモデルを追加`
- 失敗した変更を残さない。
- 不要な一時ファイル、キャッシュ、ログはコミットしない。
- **Git commit / tag / push の実行自体は、上記「Claude Code運用ルール」6節のとおりClaude Codeに実行させず、ユーザー側手動コマンドとして別途提示する。** このセクションはコミット内容・粒度・メッセージの書き方の指針であり、実行主体を規定するものではない。

## 実装品質

- Pythonは型ヒントを付ける。
- 関数は責務を小さくする。
- 入力データのバリデーションを行う。
- 例外時に原因が分かるエラーメッセージを出す。
- ファイル入出力はUTF-8を前提にする。
- 日本語テキストを扱うため、文字化けに注意する。

## テスト

- 主要処理にはテストを追加する。
- 既存テストがある場合は壊さない。
- 実装後に最低限以下を確認する。
  - サンプルJSONが読み込める
  - ブラッシュアップMarkdownが出力できる
  - Canva向け設計Markdownが出力できる
  - テストが通る

## ドキュメント

- 実装内容が変わったらREADME.mdを更新する。
- 入力形式が変わったら `docs/03_data_format.md` を更新する。
- 出力形式が変わったら `docs/04_output_spec.md` を更新する。
- タスク完了状況が変わったら `docs/05_implementation_tasks.md` を更新する。

### ドキュメント整合性の運用ルール（重要）

- コード・CLI・出力仕様・テストに変更を加えた場合は、必ず `docs/05_implementation_tasks.md` のPhase/Task進捗を確認し、必要に応じて更新する。
- 仕様変更や新機能追加を行った場合は、README.mdおよび `docs/01_requirements.md`〜`docs/04_output_spec.md` の該当箇所に矛盾がないか確認する。
- `docs/99_*` は時点スナップショット文書として扱い、既存ファイルを上書きしない。
- 新しいレビュー結果や実装状態スナップショットを残す場合は、日付またはバージョン付きの新規ファイル（例: `docs/99_review_2026-08.md`）として追加する。
- `docs/99_*` の内容は作成時点の情報であり、現行仕様の正はREADME.md・`docs/01_requirements.md`〜`docs/04_output_spec.md`・実コード・pytest結果とする。
- 作業完了報告には、`docs/05_implementation_tasks.md` を更新したか・更新不要と判断したか、および`docs/99_*` を新規追加したか・追加不要と判断したかを必ず含める。

## 禁止事項

- ユーザー承認なしに技術スタックを大きく変更しない。
- 不要に複雑な構成にしない。
- 外部API前提にしない（OpenAI API/Gamma API/Canva API等の外部LLM・API連携を前提にした実装をユーザー承認なしに追加しない）。
- 有料サービス前提にしない。ただし、ChatGPT・Claude Code・Canva・Gammaのような既存サブスク製品を、人が手動で使う運用（画面へのコピペ等）は禁止しない。
- 元教材の意図を無視して文章を大きく改変しない。
- Canvaで再現できないレイアウトを前提にしない。
- 「LLM」という言葉をChatGPT/Claude等の総称として曖昧に使わない。本プロジェクトの文脈でのLLM活用は、まずローカルLLMを指す（詳細はREADME.md「プロジェクト方針：外部API非依存・ローカルLLM移行前提」・[`docs/07_api_integration_design.md`](docs/07_api_integration_design.md)参照）。

## プロジェクト設計ルール（output構成・editable中間ファイル・source情報の扱い）

Phase 9.2までに確定した、このプロジェクトの共通設計ルールです。**今後のPhaseで変更する場合は、「基本方針」のとおり理由と影響範囲を説明して承認を得てから変更してください。** 詳細な仕様は[`docs/04_output_spec.md`](docs/04_output_spec.md)を正とし、ここではプロジェクト標準として要点のみ固定化します。

### 1. output構成

`build-all`が生成する`output/`配下は、以下の5ディレクトリに役割を分ける。

```text
output/
  editable/
    lesson_pages.json      # 正式な編集対象（再生成時にユーザーが編集するのはここのみ）
  rendered/
    page_NNN.png           # 正式な完成画像
  canva/
    canva_design.md        # 正式なCanva指示書（オプション出力。主outputではない）
  exports/
    material.md / .docx / .pdf / .pptx   # 正式な完成output
  compat/
    lesson_pages.json / canva_design.md / brushup.md / brushup.docx / brushup.pdf
                            # Phase 8以前の旧仕様との互換用output。新規利用では通常参照しない
```

各ディレクトリの役割:

- `editable/`: 再生成時にユーザーが編集する正式な中間output。
- `rendered/`: 配布・確認用の完成画像。
- `canva/`: 正式なCanva指示書。数ある完成output形式の一つであり、主outputではない。
- `exports/`: 正式な完成output（Markdown/DOCX/PDF/PPTX）。
- `compat/`: Phase 8以前の旧仕様との互換用output。`--no-compat-output`で生成自体を無効化できる。新規利用では通常参照しない。
- `output_dir`直下には、通常ユーザーが使う完成outputを置かない（`scenario/`・`review_report.md`・`imported_pages.json`・`assets/`は正式outputとの役割重複が無いため直下のまま）。

### 2. editable中間ファイルの扱い

`output/editable/lesson_pages.json`は、ユーザーが編集する正式な中間outputである。**完成画像・PDF・DOCX・PPTXを直接編集するのではなく**、以下の運用をプロジェクト標準とする。

```text
output/editable/lesson_pages.json を編集
↓
regenerate
↓
rendered/ や exports/ を再生成
```

### 3. source情報の扱い

- 元資料由来の画像・ページ画像・source assetsを落とさない。PDF・PPTX・画像などから取り込んだ元画像・ページ画像は、可能な限り`source_image`/`source_assets`（既存設計）として保持する。
- `source_page_no`は内部メタデータとして保持する。通常、購入者・受講者向けの完成output（`rendered/`・`exports/`・`canva/`・DOCX/PDF/動画シナリオ）には表示しない。制作者が確認したい場合は`review-report`コマンドを使う。

### 4. ログ出力の共通設計ルール（Phase 10.2で追加）

アプリ実行時は、プロジェクト直下の`logs/`配下に実行ログを出す。**この仕様は今後のPhaseでも維持すべき標準仕様**であり、詳細な形式は[`docs/04_output_spec.md`](docs/04_output_spec.md)「プロジェクト標準output構成」の`logs/`節を正とする。

```text
logs/
  .gitkeep
  YYYYMMDD_HHMMSS_<command>.log
```

- ログ出力対象: `build-all`・`generate`（`lesson-pages --mode generate`）・`regenerate`・`check-ocr`。個別CLI（`import-source`/`lesson-pages`の他モード/`canva`/`docx`/`pdf`/`scenario`/`canva-sync`/`wp-publish`）でも同様にログを出す。
- 失敗時（非ゼロ終了・例外発生時）にも可能な限りログを残す。
- 警告を出したうえで処理を継続する場合も、必ずログに残す（警告を出しっぱなしで記録が残らない状態にしない）。
- Git管理: `logs/`ディレクトリ自体（`logs/.gitkeep`）はGit管理対象。**ログファイル本体（`logs/*.log`）はGit管理対象外**（`.gitignore`の`logs/*`＋`!logs/.gitkeep`）。
- ZIP管理: `logs/`配下のログファイルは**ZIP対象に含める**（`input/`・`output/`は引き続きZIP対象外のまま）。

**機密情報マスク（Phase 10.2追加修正）**: `logs/*.log`はZIP対象になるため、CLI引数・標準エラー出力・エラーメッセージ等に秘密情報が混ざっていても、ログへ生値を残さない。`password`/`passwd`/`secret`/`token`/`api_key`/`apikey`/`access_key`/`access_token`/`authorization`/`bearer`/`client_secret`/`refresh_token`/`private_key`（大文字小文字を区別しない）というキー・文字列を含む値は、ログ書き出し前に`[REDACTED]`へ置換する（`src/execution_logger.py`の`mask_secrets()`。ログファイルへ書き出す直前の最終テキストに対して適用し、CLI引数・セクション内容・stderr内容・警告・エラーのいずれも漏れなくマスク対象にする）。`.env`や認証情報ファイルの中身自体をログに出力する処理は無い。

### 5. 成功判定の方針（実質失敗を正常終了扱いにしない）

Phase 10.1のOCR必須チェックと同じ考え方を、他の処理にも適用する。**「exit code 0だが目的を満たしていない」「警告は出ているが正常終了扱いでよいか曖昧」という状態を作らない。**

非ゼロ終了にすべきもの（例）:

```text
- 入力ファイル・入力ディレクトリが存在しない
- 入力ディレクトリが空、または対応ファイルが1つも無い
- 取り込み・読み込んだpagesが0件
- OCR必須モード(proofread/restructure)でOCR不能・全ページOCR結果が空
- 指定したoutput-formatの成果物が実際には生成されていない
- JSON構文エラー
```

警告のうえ処理を継続してよいもの（例）:

```text
- 一部ページだけOCR結果が空
- 一部source_image参照が無いが、テキスト出力は可能
- compat出力が無効化されている（--no-compat-output）
- フォント未指定で環境の代替フォントを使用
```

`check-ocr`は診断コマンドであり、OCR環境が不足していてもそれ自体は`exit 0`でよい（診断自体が例外で壊れた場合のみ非ゼロ終了にする）。

### 6. 今後のPhase指示文での参照方法

今後のPhase指示文では、このルールを毎回書き下す代わりに、以下のように短く参照してよい。

```text
CLAUDE_RULES.md および docs/04_output_spec.md に定義済みの
output構成・editable中間ファイル・source情報の扱い・ログ出力仕様・成功判定の方針を
維持してください。今回のPhaseでは、それらの既存設計を変更しないでください。
```

## 重要な前提

このシステムの目的は、教材画像・教材ページの内容をもとに、以下を作ることです。
